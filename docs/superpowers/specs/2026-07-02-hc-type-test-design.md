## 1. 배경 & 목표

### 행사 컨텍스트
HIGHTHON : HOMECOMING DAY는 2017년 시작된 학생 주도 해커톤 커뮤니티(11회·누적 900여 명)의
졸업생들이, 이제 현업 전문가가 되어 다시 만나는 네트워킹 밋업이다.

- **일시/장소**: 2026.07.25(토), AWS 코리아(강남 센터필드)
- **타겟**: 2004년생 이하, 일찍 IT 진로를 택한 개발자·기획자·디자이너. 주니어보다 미들/시니어 비율이 높음. 100명 한정.
- **핵심 정체성**: "같은 출발선, 다른 길" — 비슷한 추억·관점을 공유해 첫 대화부터 깊이가 다른 네트워킹.
- **세대 마커**: 이들의 고교 해커톤 시절(대략 2017~2021)은 **Copilot·ChatGPT 이전 시대**. "우리 땐 GPT도 없어서 스택오버플로우 복붙했잖아"가 이 세대만 공감하는 강력한 노스탤지어 트리거다.

### 이 기능의 목표
**바이럴 확산** — 더 많은 타겟(같은 배경의 IT 실무자)에게 행사 정보가 전파되도록 한다.
타겟이 대중이 아니라 "찾기 어려운 좁은 니치"이므로, 최고의 바이럴 루프는
**기존 졸업생이 자기 결과물을 공유 → 인스타/오픈채팅에서 비슷한 배경의 친구가 보고 → "나도 하이톤이었는데" → 유입**이다.

### 성공 지표 (PoC)
- 테스트 완료 수 / 결과 공유(이미지 저장·링크 복사) 클릭 수
- 결과 페이지 → 티켓 예약(`/homecoming/ticket`) 전환 클릭 수
- 유입 경로: 공유 링크(유형별 OG 카드) 클릭으로 들어온 방문 비율

## 2. 채택한 방향 (결정 기록)

브레인스토밍에서 확정된 갈림길:

1. **공유 카드 + 성향 테스트를 하나로 융합** → 테스트가 메인, 결과 카드가 산출물.
   (테스트 자체가 바이럴 엔진: "나 XX 유형이래 ㅋㅋ 너는?" 포맷이 태그·공유를 부른다.)
2. **유형 축 = 하이톤 시절 노스탤지어/밈 캐릭터** (커리어 성향·네트워킹 스타일이 아닌 "그 시절 나는 어떤 팀원?").
   행사 대상이 "하이톤을 기억하는 성인"이라 노스탤지어 축이 오히려 타겟과 정확히 맞물린다.
3. **완전 독립형** — 등록 없이 누구나 진입, 결과 끝에서 예약 CTA. 바이럴 도달 극대화.
4. **직군 균형** — 개발자·기획자·디자이너가 모두 "이건 내 얘기"라고 느끼도록 유형을 직군별로 배분.
5. **pre-AI 정서**를 유형·문항에 반영. 단, 하이톤 참가 여부와 무관하게 공감되도록 "그 시절 해커톤/개발 좀 했던 우리" 톤으로 넓게.

## 3. 바이럴 루프 (사용자 흐름)

```
인스타·오픈채팅 링크
      ↓
/homecoming/type  (랜딩: 등록 불필요, 누구나)
      ↓
10문항  (그 시절 밈 상황 기반, 직군 중립 질문)
      ↓
결과: 16유형 중 하나 + 캐릭터 카피 + 궁합(찰떡 팀 / 물과 기름)
      ↓
결과 화면 = 공유 카드
   ├─ "결과 공유" → 이미지 저장 / 링크 복사 (유형별 OG 카드가 리치 프리뷰)
   └─ "이 유형들 실제로 만나러 가기" → /homecoming/ticket (예약)
      ↓
친구가 공유 링크 클릭 → 유형별 OG 카드 프리뷰 → 유입 → 반복
```

핵심: **공유 = 링크**. 결과 URL(`/homecoming/type/result/[typeId]`)이 곧 유형별 OG 카드를 생성하므로,
링크만 뿌려도 인스타·카톡·X에 리치 프리뷰가 붙는다.

## 4. 유형 시스템

### 4.1 채점 방식
- **순수 클라이언트 사이드 채점** (DB·백엔드 불필요).
- 각 문항의 각 선택지가 하나 이상의 유형에 **가중치**를 부여 → 총점 최고 유형이 결과.
- 동점 시 tie-break 규칙: 선언된 유형 배열 순서(또는 문항 가중치 우선순위) 기준으로 결정론적 선택.
- 장점: 문항-유형 매핑이 유연하고, 문항/유형을 상수 데이터로만 조정 가능.

## 5. 문항 설계 원칙

- **직군 중립**: "무슨 프레임워크 써?" 같은 개발 편향 질문 금지. 개발·기획·디자인 누구나 겪은 상황으로.
  - 예: "발표 준비할 때 나는?", "새벽 3시 팀 분위기는?", "에러/막힘을 만나면?", "아이디어 회의에서 내 역할은?", "데모 30분 전 우리 팀은?"
- **pre-AI 정서**: 선택지에 GPT/Copilot이 애초에 없다. "스택오버플로우 정독 / 일단 다 지우고 재작성 / 옆자리 붙잡고 물어봄 / 프린트문 도배" 식.
- **10문항** — 16유형을 가르려면 판별력이 필요. 8문항이면 결과가 뭉개질 수 있어 10문항으로 시작.
- 각 선택지는 1~2개 유형에 가중치를 준다(상수 데이터로 관리).

## 6. 기술 설계

> ⚠️ 이 저장소의 Next.js는 학습 데이터와 API가 다를 수 있음(AGENTS.md). 구현 전
> `node_modules/next/dist/docs/`의 관련 가이드와 기존 `src/app/homecoming/opengraph-image.tsx`
> 패턴을 반드시 확인할 것. 아래는 설계 의도이며, 실제 API 시그니처는 구현 계획 단계에서 검증한다.

### 6.1 라우트 구조 (App Router)
```
src/app/homecoming/type/
  page.tsx                         # 랜딩 + 인트로 ("시작하기")
  quiz/page.tsx  (or client comp)  # 문항 진행 (클라이언트 상태)
  result/[typeId]/
    page.tsx                       # 결과 화면 (공유/CTA)
    opengraph-image.tsx            # 유형별 동적 OG 카드
```
- 결과는 URL 경로(`typeId`)로 표현 → 공유 링크가 곧 유형별 리치 프리뷰.
- 채점은 클라이언트에서 수행 후 `/result/[typeId]`로 이동. `typeId` 유효성 검증(미존재 → 랜딩으로 리다이렉트/안내).

### 6.2 데이터 (상수 파일)
`src/lib/hcTypeTest.ts` 하나에 몰아서 관리:
- `HC_TYPES`: 유형 16종 (id, 이름, 직군, 한 줄, 상세, best/worst, 카드 색/이모지 모티프)
- `HC_QUESTIONS`: 문항 + 선택지 + 선택지별 유형 가중치
- `scoreAnswers(answers)`: 가중치 합산 → 최고 유형 id 반환 (결정론적 tie-break 포함)
- 타입 정의(`HcType`, `HcQuestion`, `HcChoice`). `any` 금지, union 유지.

### 6.3 공유 이미지 — 2종
1. **링크 프리뷰용 OG (가로 1200×630)** — 카톡·X·링크 언펄용.
   - `result/[typeId]/opengraph-image.tsx`가 `params.typeId`로 유형을 찾아 렌더. 기존 `opengraph-image.tsx` 패턴 재사용.
2. **스토리 공유용 카드 (세로, 1080×1920 또는 1080×1350)** — 인스타 스토리에 저장·업로드용.
   - 결과 페이지의 "🖼 저장" 버튼이 이 세로 카드를 이미지로 저장.
   - 구현: 별도 `route.ts`(next/og로 세로 사이즈 렌더) 이미지 URL 다운로드 방식을 우선 검토(클라이언트 캡처보다 서버 렌더가 폰트·레이아웃 안정적). 구현 단계에서 난이도 확인 후 확정.
- **카드 요소(양쪽 공통)**: 유형 이름, 직군 태그, 이모지 모티프, 한 줄 카피, 궁합(찰떡/상극), 유형별 색, "HIGHTHON : HOMECOMING DAY" 브랜딩, 날짜(2026.07.25), 인스타 핸들.

### 6.4 재사용 자산
- UI 톤: 기존 Tailwind 토큰(`ink`/`paper`/`line` 등), `Reveal`, `TrackedLink`, `HcNavbar/HcFooter`.
- 전환 측정: 결과 페이지 CTA·공유 버튼에 Vercel `track` 이벤트
  (`type_test_start`, `type_test_complete`, `type_test_share`, `type_test_reserve_click`).

### 6.5 개인정보/저장
- PoC는 **데이터 저장 없음**. 응답·결과를 서버에 남기지 않는다(클라이언트 채점 → URL 이동).
- 향후 확장(유형 분포 집계, 등록 시 유형 연동)은 별도 스펙.

### 6.6 UI/레이아웃 결정 (와이어프레임 확정)
- **모바일 퍼스트**. 데스크탑도 **동일 모바일 UI를 그대로** 사용하되 **최대 너비 600px, 가운데 정렬**.
- **인트로**: 다크 히어로(검정 배경 · 기존 `HcHero` 톤 계승) + 큰 타이포("무슨 스타일"은 외곽선 stroke 강조) + 하단에 **유형 카드 마퀴**(살짝 기울어진 컬러 카드가 옆으로 흐르는 동적 프리뷰) + "테스트 시작" 버튼.
- **문항(×10)**: 다크 심플. 상단 10칸 진행바 + `Q{n}/10` + 질문 + **아웃라인 4지선다**. **선택 즉시 다음 문항**(별도 '다음' 버튼 없음).
- **결과**: 리빌 + 섹션(다크). 유형명 리빌(외곽선 강조) + 이모지 모티프 + 상세 설명 → "그때의 궁합"(찰떡/상극) → 하단 **공유 카드 미리보기 썸네일** → 버튼 3개(🖼 이미지 저장 · 🔗 링크 복사 · 예약 CTA).
- **색 시스템**: 유형별 색(노랑·블루·핑크·그린·뉴트럴 등)을 카드·포인트에 사용. 상수 데이터의 유형별 `color` 필드로 관리.

## 8. 열린 질문
- 문항 수: 16유형 판별을 위해 10문항으로 시작. 완주율이 낮으면 8로 축소 검토.
- "이미지 저장" UX: 세로 카드를 next/og 서버 렌더 → URL 다운로드 방식 우선. 구현 난이도 확인 후 확정.
- 세로 카드 사이즈: 1080×1920(풀 스토리) vs 1080×1350(피드/스토리 겸용) — 구현 시 택1.
- 유형별 이모지 모티프 vs 간단 일러스트 — PoC는 이모지+색으로 시작, 이후 일러스트 승격 가능.
